# Top10 Mapping Reporter

OWASP Top 10 2025 기준으로 **명확하게 확인 가능한 보안 점검 결과**를 수집하고,  
그 근거를 함께 매핑하여 **리포트로 정리하는 도구**입니다.

이 프로젝트는 취약점을 자동으로 공격·탐지하는 스캐너가 아니라,  
**점검 결과를 근거 중심으로 정리하고 설명하는 Reporter**를 목표로 합니다.

---

## 🔍 Why this project?

OWASP Top 10을 공부하면서 다음과 같은 고민이 들었습니다.

- OWASP Top 10의 개념은 이해했지만,  
  **실제로 무엇을 점검하고 어떻게 보고해야 하는지**는 잘 와닿지 않았습니다.

- 자동 취약점 스캐너는 결과는 많지만,  
  **왜 이게 문제인지 설명하기 어렵고**, 오탐도 적지 않았습니다.

- 포트폴리오 관점에서도 단순히 “스캐너를 돌려봤다”는 것보다  
  **점검 기준과 판단 과정을 설명하는 것이 더 중요하다고 느꼈습니다.**

그래서 이 프로젝트는 공격 페이로드를 사용하는 방식 대신,  
**관찰 가능하고 재현 가능한 사실과 근거를 남기는 것**에 집중합니다.

---

## 🎯 Project Goal

- HTTP 요청/응답을 기반으로 **명확히 확인 가능한 보안 설정을 점검**

- 각 점검 항목을 **OWASP Top 10 2025 카테고리에 근거와 함께 매핑**

- 결과를 사람이 읽기 쉬운 **HTML / Markdown 리포트** 형태로 출력

- 모든 판단에는 **재현 가능한 증거(Evidence)**를 함께 남김

---

## 📦 What this tool does (MVP scope)

### Included
- HTTP 보안 헤더 설정 확인
- Cookie 속성(Secure, HttpOnly, SameSite) 식별
- 서버 정보 노출 여부 확인
- robots.txt / sitemap.xml 존재 여부 확인

### Not included
- SQL Injection, XSS, SSRF 등 공격 페이로드 기반 탐지
- 외부 실서비스에 대한 무단 점검
- 취약점 “확정” 판정

## 📊 결과 상태(Result Status)

이 도구의 점검 결과는 다음과 같은 상태 값으로 표현됩니다.  
모든 상태는 **관찰 가능한 근거를 기준으로 한 결과 표현**이며, 취약점을 확정하는 판단이 아닙니다.

| 상태 | 설명 |
|---|---|
| 확인됨 (PASS) | 보안 설정이 정상적으로 확인된 상태 |
| 설정 미흡 (WEAK) | 설정은 존재하지만 보안 관점에서 충분하지 않은 상태 |
| 미설정 (MISSING) | 관련 보안 설정이 확인되지 않은 상태 |
| 식별됨 (OBSERVED) | 보안상 의미를 가질 수 있는 요소가 식별되어 주의가 필요한 상태 |
| 확인불가 (NOT_TESTED) | 환경적·기술적 이유로 판단에 필요한 근거를 확보하지 못한 상태 |

### 식별됨(OBSERVED)에 대하여

**식별됨**은 “취약하다”는 의미가 아닙니다.  
OWASP Top 10 매핑 기준에 따라, **보안상 의미를 가질 수 있어 추가적인 검토가 필요하다고 판단된 요소**를 나타냅니다.

예를 들어 다음과 같은 경우에 사용됩니다.

- 서버 정보 또는 버전이 응답 헤더에 노출된 경우
- robots.txt 또는 sitemap.xml에서 관리 경로가 식별된 경우
- 세션 쿠키로 보이는 값이 확인되었으나, 의미를 단정할 수 없는 경우

## 🧭 How it works (High-level)

1. 사용자가 대상 URL을 입력
2. 시스템이 HTTP 요청을 보내 응답을 수집
3. 체크리스트 기반 점검 수행
4. 점검 결과를 OWASP Top 10 2025 항목과 매핑
5. 근거(Evidence)와 함께 리포트 생성

## ⚠️ 주의 사항 (Disclaimer)

이 프로젝트는 **로컬 데모 환경 또는 명확히 허가된 테스트 환경**을 대상으로 합니다.

외부 서비스에 대해 무단으로 공격하거나 스캔하기 위한 도구가 아닙니다.

실제 운영 환경에 적용할 경우에는 반드시 관리자 또는 서비스 소유자의 사전 허가를 받아야 합니다.
